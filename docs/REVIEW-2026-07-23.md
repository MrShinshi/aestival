### 1.3 SIGINT/SIGTERM 信号处理器为空操作

**文件**: [app/main.cpp](../app/main.cpp#L207-L209)

```cpp
std::atomic<bool> shutdown{false};
std::signal(SIGINT, [](int) { /* handled via polling */ });
std::signal(SIGTERM, [](int) {});
```

`shutdown` 原子变量在任何地方都未设置为 `true`。两个信号处理器捕获信号后空操作，抑制默认终止行为。systemd 发送 SIGTERM 超时后将 SIGKILL，导致 SQLite WAL 文件、WebSocket 连接可能损坏。

**修复方案**:
```cpp
std::signal(SIGINT, [](int) { shutdown.store(true); });
std::signal(SIGTERM, [](int) { shutdown.store(true); });
// 主循环检查 shutdown
```

### 1.4 分离线程中的 use-after-free

**文件**: [core/src/agent_registry.cpp](../core/src/agent_registry.cpp#L414-L426)

```cpp
auto* inst_ptr = &inst;
std::thread([inst_ptr]() {
    std::this_thread::sleep_for(std::chrono::seconds(3));
    if (inst_ptr->controller) { // 若 agent 已销毁则为悬空指针
        inst_ptr->controller->notify_startup();
    }
}).detach();
```

若 agent 在这 3 秒内被 `remove_agent` 或 `stop_all` 销毁，`inst_ptr` 成为悬空指针。

**修复方案**: 使用 `std::weak_ptr` 进行生命周期追踪，或将通知逻辑整合到现有的 session 连接回调中。

## 🟠 高优先级

### 2.1 Web UI 无认证自动登录

**文件**: [webui/backend/src/auth.ts](../webui/backend/src/auth.ts#L15-L31)

OAuth 被移除后（提交 `ff8e374`），所有 Web UI 端点无需认证即可访问：
- `GET /api/ui/auth/me` 无条件返回 `{ user: 'admin' }`
- `GET /api/ui/auth/token` 无条件签发 24h 有效期的 admin JWT

任何能访问 Web UI 的人（直接或通过反向代理）都拥有对 bot 的**完全管理控制权**（启动/停止/删除 agent、修改配置）。

**修复方案**:
1. 若仅 localhost 使用：添加 IP 白名单或 `auth_basic`
2. 若通过 Nginx 暴露：配置 Nginx `auth_basic` 或 IP 限制
3. 考虑恢复最小化认证（如单用户密码/TOTP）

### 2.2 SQLite 连接泄漏

**文件**: [webui/backend/src/conversations.ts](../webui/backend/src/conversations.ts#L35-L96)

列表端点和详情端点中，若 `db.prepare()` 或 `stmt.all()` 抛出异常，`db.close()` 永远不会被调用：

```typescript
const db = openDb('default');
const stmt = db.prepare(`...`); // 若抛出，db 永不关闭
const rows = stmt.all(limit);
db.close(); // 仅成功路径执行
```

长时间运行的 Web UI 进程可能耗尽文件描述符。

**修复方案**: 使用 `try/finally` 或辅助包装器，确保 `db.close()` 在所有路径上被调用。

---

### 2.3 Conversations 端点的路径遍历

**文件**: [webui/backend/src/conversations.ts](../webui/backend/src/conversations.ts#L17-L20)

```typescript
const paths = [
  `${CONTEXTS_BASE}/${agentId}/conversations.db`, // agentId 来自 req.query
  `${CONTEXTS_BASE}/conversations.db`,
];
```

`agentId` 来自查询参数，未做清理直接拼接到路径。攻击者可传入 `../../etc/passwd` 遍历目录。

**修复方案**: 对 `agentId` 做清理——移除 `..`、路径分隔符、空字节，或使用白名单字符集。

### 2.4 管理 API 无输入验证

**文件**: [core/src/management_api.cpp](../core/src/management_api.cpp#L174-L235)

`handle_agents_create` 和 `handle_agent_config` 直接解析 JSON 为 `agent_config`，不验证：
- Agent ID 格式（可含路径分隔符、控制字符）
- 字符串字段长度
- `storage_dir` 中的路径遍历（`../../../etc`）
- 负值或越界数值

**修复方案**: 添加 ID 字符白名单、路径清理、字符串长度限制。

### 2.5 SSH 私钥无保证清理

**文件**: [.github/workflows/ci.yml](../.github/workflows/ci.yml#L173-L198)

SSH 私钥被 `echo` 写入磁盘，仅在步骤末尾 `rm -f`。若步骤失败，密钥残留；runner 可能被复用，密钥可泄露给其他 actor。

**修复方案**: 使用 `trap` + `finally` 确保清理；或使用 `ssh-agent` 非持久化注入。

### 2.8 前端无 fetch 超时与 401 拦截

**文件**: [webui/frontend/src/lib/api.ts](../webui/frontend/src/lib/api.ts#L8-L17)

`request()` 函数：
- 不使用 `AbortController` 设置超时——停滞请求永久挂起
- 不处理 401 响应——认证过期不触发重定向

**修复方案**:
```typescript
const controller = new AbortController();
const timeout = setTimeout(() => controller.abort(), 30_000);
const res = await fetch(url, { ..., signal: controller.signal });
clearTimeout(timeout);
if (res.status === 401) { /* 重新认证 */ }
```

### 3.1 SSRF 保护不完整

**文件**: [core/src/agent_controller.cpp](../core/src/agent_controller.cpp#L386-L452)
